## 1. Motivation

The nature of merge conflicts produced by human developers has been extensively characterized by Ghiotto et al. (TSE 2020), who examined more than 960,000 merges across 2,731 open-source Java projects and established a multi-dimensional empirical portrait of conflicting chunks: number of chunks per failed merge, chunk size in lines of code, language constructs involved, developer decisions (V1, V2, CC, CB, NC, NN), association patterns, and difficulty levels. This portrait has become the substrate on top of which virtually all subsequent research on automated merge conflict resolution â€” from structured and semi-structured techniques to recent learning-based approaches â€” has been conceived and calibrated.

With the massive adoption of autonomous coding agents â€” Codex, Devin, Copilot, Cursor, and Claude Code had collectively produced more than 930,000 pull requests across 116,000 repositories by August 2025, as documented by the AIDev dataset â€” a fundamental empirical gap emerges: **we do not know whether the nature of the conflicts generated by agent-authored PRs resembles the nature of conflicts generated by humans.** Three consequences make this question urgent. First, existing automated resolution techniques were designed on top of a characterization that may not correspond to the new regime, and their validity for agent conflicts is an untested assumption. Second, the AIDev literature itself (Pradel et al., Research Direction #4) lists the treatment of merge conflicts by coding agents as an open research direction. Third, every conflict that the agent does not resolve falls back to a human developer, so characterizing the nature of these conflicts is a prerequisite for any effort aimed at reducing the human burden imposed by agent-driven development.

By replicating and adapting Ghiotto et al.'s analytical framework over the universe of agent-authored PRs cataloged in AIDev, this study contributes the first systematic empirical characterization of the nature of merge conflicts in that regime, and contrasts it, via the literature, with the consolidated human portrait. The results provide the empirical foundation required to reassess the applicability of existing resolution techniques, to identify patterns specific to the agent-authored regime, and to guide future research on developer assistance and on the agents themselves with respect to the code integration workflow.

---

## 2. Main Objective

To empirically characterize the nature of merge conflicts occurring inside pull requests authored by autonomous coding agents â€” along structural dimensions (number and size of conflicting chunks, size of the resolution), resolution-strategy dimensions (which of the canonical decisions V1/V2/CC/CB/NC/NN was made), and actor dimensions (who performed the resolution) â€” and to contrast that characterization, via the literature, with the consolidated portrait of human-authored merge conflicts, establishing the empirical foundation required for the evolution of conflict-resolution techniques in the era of coding agents.

---

## 3. Research Questions

**RQ1 â€” Structural nature.** In merge commits that occur inside agent-authored PRs, what are the distributions of (i) the number of conflicting chunks per failed merge, (ii) the size in lines of code of version 1 (`v1`) of each chunk, (iii) the size in lines of code of version 2 (`v2`) of each chunk, and (iv) the size in lines of code of the resolution chosen for each chunk?

**RQ2 â€” Resolution strategies.** How are resolution decisions distributed across conflicting chunks (V1, V2, CC, CB, NC, NN, Imprecise), and how does this distribution compare to the one reported by Ghiotto et al. for human-authored merges?

**RQ3 â€” Resolver.** Who resolves these conflicts â€” the coding agent itself or a human collaborator?

**Cross-cutting axis.** For every RQ in which it is meaningful, results are stratified by (a) coding agent (Codex, Devin, Copilot, Cursor, Claude Code), (b) programming language (using the repository's `language` field), and (c) PR task type (using the `pr_task_type` classification already provided by AIDev: `feat`, `fix`, `docs`, etc.).

The study deliberately excludes, relative to Ghiotto et al.'s framework, the analyses that depend on Java-specific language-construct parsing (A3, A4, A7) and the difficulty-ratio aggregation (A6). The rationale is twofold: (i) AIDev covers multiple programming languages with no uniform construct vocabulary, and (ii) including construct-level analysis would inflate the scope beyond what is feasible for this first characterization paper. Those dimensions are left as future work.

## 4. Dataset and Scope

### 4.1 Source

The full AIDev dataset (`all_pull_request.parquet` + `all_repository.parquet`), containing roughly 930,000 PRs across 116,000 repositories as documented in the AIDev paper. The restricted AIDev-pop tier (stars > 100, ~33,600 PRs / ~2,807 repositories) remains available as an optional run mode (`--pop-only`) for faster iteration and for cross-checking against the enriched tables that AIDev only publishes at the pop tier (`pr_task_type`, `pr_timeline`, `pr_comments`, `pr_reviews`, `pr_commit_details`, `issue`, `related_issue`).

AIDev publishes the per-PR commit inventory (`pr_commits.parquet`) only for AIDev-pop, which would bound the study at pop if this file were the only source. To unlock the full AIDev tier, a companion pipeline `extract_pr_chronology.py` fetches `refs/pull/*/head` for every repository in the scope and enumerates, per PR, the commits between the fork-point with the base branch and the PR tip. The output is written to `data/pr_chronology/pr_commits.parquet` with schema `(pr_id, repo_full_name, pr_number, sha, author_date, commit_index, commit_count)` and supersedes AIDev's file; it also carries the full chronology of a PR (the `commit_index` column), which lets us correlate the order of the commits with their outcome.

### 4.2 Target universe

All PRs in the selected scope (full AIDev by default, AIDev-pop under `--pop-only`), regardless of final state (`open`, `closed` without merge, or `merged`). Restricting the universe to merged PRs would bias the sample toward PRs whose history was rewritten via rebase-and-merge or squash-and-merge to keep a linear base-branch history â€” which would drop exactly the intermediate merge commits that carry the conflicts we want to study. Including PRs of all states lets us characterize the full distribution of conflict-bearing merges.

### 4.3 Unit of analysis

An **internal merge commit** of a PR â€” that is, a commit referenced by the PR (`pr_commits.sha`) that has exactly two parents. Single-parent commits (ordinary commits or fast-forwarded rebases) are excluded because they cannot produce textual conflicts. Octopus merges (three or more parents) are rare in practice and are excluded from the analysis set to keep the methodology aligned with Ghiotto et al., who studied three-way merges only; each octopus is recorded in `extraction_errors.parquet` with `error_type="octopus_merge"` so the audit count is preserved.

The **final merge commit** that integrates the PR into the base branch (created automatically by GitHub when a PR is merged via "Create a merge commit") is excluded from the main analysis, because GitHub's mergeability check gates the PR and makes those merges almost always conflict-free. Detection is performed on the commit message: any two-parent commit whose subject matches `^Merge pull request #\d+` (case-insensitive) is filtered out and recorded in `extraction_errors.parquet` with `error_type="pr_integration_merge"`. A one-off audit over the full universe will report the fraction of these integration merges that did produce a conflict, for completeness.

The analysis granularity inside each internal merge commit is the **conflicting chunk**, consistent with Ghiotto et al.

## 5. Methodology

### 5.1 Identifying internal merge commits

The pipeline runs in two passes.

**Pass A â€” chronology (`extract_pr_chronology.py`).** For every repository in scope, the script adds the remote `origin` and runs `git fetch origin +refs/pull/*/head:refs/pull/*/head`, which populates a local namespace of PR head refs without paying the GitHub API rate limit. For each PR of the repository, it computes `git merge-base refs/pull/<N>/head HEAD` to find the fork-point with the base branch and runs `git log <base>..refs/pull/<N>/head --reverse --format=%H|%aI` to enumerate every commit introduced by the PR, in chronological order. The output is persisted as `data/pr_chronology/pr_commits.parquet` with schema `(pr_id, repo_full_name, pr_number, sha, author_date, commit_index, commit_count)`. PRs whose `refs/pull/<N>/head` is not present on the remote at fetch time are recorded in `data/pr_chronology/errors.parquet` (force-deleted branches, very old PRs whose refs were garbage-collected by GitHub, private forks that have been deleted).

**Pass B â€” merge extraction (`extract_aidev_nature.py`).** For each repository in scope, a bare clone is created locally. For each PR, the set of candidate SHAs is read from the chronology produced in Pass A; this is the single source of truth for "commits belonging to a PR" and is not supplemented by any `git log` heuristic. PRs absent from the chronology (chronology has not yet processed that repository, or the PR has zero own commits past the fork-point) are logged as `error_type="no_sha_for_pr"` and skipped. For every candidate SHA, `git cat-file -p <sha>` is executed on the bare clone to retrieve its parents. Commits with exactly two parents are candidate internal merges; those whose subject matches the GitHub integration-merge pattern (Â§4.3) are filtered out at this point; the rest are kept. Commits with three or more parents are recorded as octopus merges and excluded; commits with one parent are discarded. SHAs that are no longer reachable in the clone (typical of force-pushed / rebased PR branches) are logged as `error_type="sha_not_in_repo"` and skipped. The bare clone is retained for the duration of the shard that processes its PRs and removed at the end of the shard to bound disk usage.

The two passes intentionally live in separate scripts. Pass A has an unavoidable network-bound step (one `git fetch` per repository) and its output is idempotent; once produced, the extraction of Pass B can be replayed arbitrarily many times without hitting the network.

### 5.2 Detecting conflicts per merge

For every internal two-parent merge `M` with parents `P1` and `P2`, the pipeline replays the three-way merge of `P1` and `P2` using their `git merge-base` as the common ancestor. The replay reuses the procedure already implemented in `collect.py`:

1. Compute `B = git merge-base P1 P2`.
2. For each file `f` whose blob differs between `P1:f`, `P2:f`, and `B:f`, materialize the three blobs and invoke `git merge-file --diff3` to produce a merged output with diff3 conflict markers.
3. Parse the output with the existing `MERGE_TREE_CONFLICT_REGEX` / `DIFF3_CHUNK_REGEX` to extract each conflicting chunk as the triple `(v1, base, v2)`.
4. If no chunk is produced, the merge is recorded as conflict-free and no further analysis is run on it.

### 5.3 Extracting the resolution per chunk

For each conflicting chunk identified in step 5.2, the corresponding region in the actual merge commit `M` has to be isolated. This is necessary because `M` also contains unrelated edits (merge-time adjustments, formatter output, etc.) and only the region that corresponds to each chunk should be compared against the strategies. The LOCALIZERESREGION routine from `collect.py` (`find_resolution`) handles this by matching context lines surrounding the chunk against the tree of `M`. The output is the per-chunk tuple `(v1, base, v2, resolution)`.

### 5.4 Classifying the resolution strategy

`identify_resolution(v1, v2, resolution)` from `extract_resolution_strategies.py` is applied to each chunk. The function returns one of seven labels used in this study:

- `V1` â€” the resolution equals `v1`.
- `V2` â€” the resolution equals `v2`.
- `CC` (Concatenation) â€” the resolution equals `v1 + v2` or `v2 + v1` (the existing function distinguishes `ConcatV1V2` and `ConcatV2V1`; both are folded into `CC` for comparison with Ghiotto et al.).
- `CB` (Combination) â€” the resolution interleaves lines from `v1` and `v2` without introducing new lines.
- `NC` (New Code) â€” the resolution contains lines that appear in neither `v1` nor `v2`.
- `NN` (None) â€” the resolution is empty.
- `Imprecise` â€” LOCALIZERESREGION could not isolate a resolution region with confidence; reported as its own bucket to avoid silent misclassification.

### 5.5 Size metrics

For each chunk, the pipeline records `v1_loc`, `v2_loc`, `base_loc`, and `resolution_loc`, computed as the number of non-blank lines after trimming whitespace (consistent with Ghiotto et al.'s treatment, `remove_empty_lines` in `extract_resolution_strategies.py`). `resolution_loc` is the new metric this study adds relative to Ghiotto's A2.

### 5.6 Identifying the resolver

The author of the internal merge commit `M` (field `pr_commits.author`, which stores the login when available and the name otherwise) is cross-referenced with the set of agent-account signatures known to AIDev. The AIDev paper (Table 2) documents the search queries used to scope each agent's PRs, and these queries expose the bot/account login per agent (for example, `devin-ai-integration[bot]`, `copilot-swe-agent[bot]`, etc.). We compile the definitive mapping directly from the `pull_request.parquet` field `user`, grouped by `agent`, which yields the set of account signatures each agent uses. A merge commit authored by an account in that set is labeled `agent-resolved`; otherwise it is labeled `human-resolved`.

### 5.7 Analyses per research question

**RQ1** â€” For each of `#chunks per merge`, `v1_loc`, `v2_loc`, and `resolution_loc`, we report the distribution (histogram, median, mean, standard deviation, selected percentiles) over the full agent universe and over the cross-cutting strata. Visualizations follow Ghiotto et al.'s figures to enable direct visual comparison.

**RQ2** â€” We report the relative frequency of each resolution label (V1, V2, CC, CB, NC, NN, Imprecise) at the chunk level, globally and per stratum. The comparison with Ghiotto is numerical and textual (no empirical human cohort is collected in this study, per scope decision). The discussion explicitly addresses how the `Imprecise` bucket, which does not exist in Ghiotto et al., affects comparability.

**RQ3** â€” We report the distribution of the resolver label at the merge-commit level (`agent-resolved`, `human-resolved`).

## 6. Extraction Pipeline

### 6.1 Stages

**Stage 0 â€” Universe preparation.** Load `all_pull_request.parquet`, `all_repository.parquet` from `AIDev/` (or `pull_request.parquet` + `repository.parquet` under `--pop-only`), the chronology from `data/pr_chronology/pr_commits.parquet`, and â€” when available at the AIDev-pop tier â€” `pr_task_type.parquet`. Build an in-memory frame keyed by `(repo_full_name, pr_id, commit_sha)` with `agent`, `language`, `pr_task_type`, `state`, `merged_at` attached. Persist as `data/nature_of_agent_conflicts/universe.parquet`.

**Stage 1 â€” Bare cloning, per-shard.** Shard repositories by the low bits of a hash of `repo_full_name` into groups sized for the worker pool (see Â§6.2). Each shard clones its repositories with `git clone --bare --filter=blob:none` into a local scratch directory, processes all PRs belonging to those repositories, and removes the clones at the end of the shard to bound disk usage.

**Stage 2 â€” Merge-commit enumeration.** For each PR in the shard, call `git cat-file -p <sha>` on each SHA from `pr_commits` of that PR to retrieve the parent list. Classify each commit as `regular` (one parent), `internal-merge` (two parents), or `octopus` (three or more parents). Persist the merge inventory as `internal_merges.parquet` with schema `(pr_id, merge_sha, parent1_sha, parent2_sha, author, committer, committed_at)`.

**Stage 3 â€” Conflict replay.** For each row in `internal_merges.parquet`, compute `git merge-base parent1 parent2`, enumerate files that differ across the three blobs, run `git merge-file --diff3`, and parse chunks. Emit `conflict_chunks.parquet` with schema `(pr_id, merge_sha, file_path, chunk_index, v1, base, v2, v1_loc, v2_loc, base_loc)`.

**Stage 4 â€” Resolution localization and extraction.** For each row in `conflict_chunks.parquet`, apply LOCALIZERESREGION against the tree of `merge_sha` for `file_path` to retrieve the `resolution` string. Emit `resolved_chunks.parquet` extending the previous schema with `(resolution, resolution_loc, localized_ok)` where `localized_ok` is false for rows that fall into `Imprecise`.

**Stage 5 â€” Strategy classification.** Apply `identify_resolution(v1, v2, resolution)` and emit `classified_chunks.parquet` adding `(strategy)`.

**Stage 6 â€” Resolver attribution.** Build the agent-account signature map from `pull_request.parquet` grouped by `agent`, cross-reference `internal_merges.parquet.author` against it, and emit `resolver_labels.parquet` with `(pr_id, merge_sha, resolver_type)` where `resolver_type` is one of `agent` / `human`.

**Stage 7 â€” Analysis.** Notebooks joining the parquet outputs, producing the per-RQ tables and plots.

Stages 0 through 6 are reproducible; the analysis notebooks are the only manual step beyond pipeline configuration.

### 6.2 Parallelization

Parallelism is granular at the repository level, not at the PR level, because colocating PRs of the same repository on the same worker amortizes the bare clone cost. The compute budget is 5 days Ă— 32 cores â‰ˆ 3,840 core-hours. A concurrent `multiprocessing.Pool` with 32 workers processes one shard each; a shard is sized so that its clones plus scratch fit within a conservative per-worker disk budget (target: 10 GB per worker, total 320 GB). A lockfile in the scratch directory prevents two workers from cloning the same repository.

### 6.3 Output directory layout

```

  pr_commits.parquet                    # (pr_id, repo_full_name, pr_number, sha,

Each chunk-level table is de-duplicated on its natural key â€” `(repo_full_name, merge_sha, file_path, chunk_index)` for `conflict_chunks` / `resolved_chunks` / `classified_chunks`, and `(repo_full_name, pr_id, merge_sha)` for `internal_merges` â€” at the JSONLâ†’Parquet aggregation step, so that PRs that legitimately share a commit (force-pushed, re-opened, cherry-picked) do not inflate downstream counts.

## 7. Computational Budget

The expected processing load is dominated by three operations: cloning bare repositories (once per repository), walking each PR's SHAs via `git cat-file` (cheap, O(SHAs) per PR), and replaying merges with `git merge-file` plus localizing resolutions (the dominant cost, concentrated on the subset of PRs that actually contain internal merges).

Back-of-the-envelope sizing for full AIDev (AIDev-pop figures in parentheses):

- ~116,000 repositories (~2,807). Average bare-clone footprint with `--filter=blob:none` ~30â€“50 MB. The shard-rotating budget (clone a shard, process it, delete, move on) keeps peak disk usage within the 320 GB budget regardless of total repositories.
- ~930,000 PRs (~33,600). Estimated 10â€“30 % contain at least one internal merge commit. Average of 1â€“3 internal merges per such PR yields on the order of 140,000â€“840,000 merge replays (5,000â€“30,000 for AIDev-pop alone).
- Per-replay cost: ~5â€“15 seconds (dominated by `merge-file` on touched files and LOCALIZERESREGION). Total sequential: thousands of hours for full AIDev. On 32 cores, wall-clock is dominated by the chronology fetch of Pass A and by cloning; the 5-day budget holds for AIDev-pop easily and for full AIDev requires sampling (Â§10) or distributing Pass B across multiple machines. The incremental tracker (`processed_repos.txt`) lets runs be resumed and shared across machines.

The numbers above are estimates to be validated during Stage 0, which produces the universe frame from which actual counts of PRs and SHAs per repository become known before any replay is executed.

## 9. Threats to Validity

**Internal validity â€” Localization errors.** LOCALIZERESREGION can fail to isolate the correct region when context lines around a conflicting chunk have been edited in the merge commit itself. Every failure is recorded in the `Imprecise` bucket so it is visible in the results rather than silently misclassified.

**Internal validity â€” Agent-account mapping.** Some agents change their account login over time, and some PRs flagged as agent-authored by AIDev may have been resolved by a human using the agent account (or vice versa). The `resolver_type` label therefore reflects account identity, not intent; the discussion will make this explicit.

**Construct validity â€” Replay vs. historical merge behavior.** `git merge-file --diff3` replays a three-way merge using the default Git strategy. Some projects use custom merge drivers (e.g., for generated files, binaries, or language-specific semantic merge), and in those cases the replay may produce conflict markers where the actual project pipeline would not have. This is the same limitation Ghiotto et al. acknowledge for their large-scale automated analysis.

**Construct validity â€” Coverage of "resolution".** Our unit of analysis is the internal merge commit, which captures the case where the agent (or a human) pulls the base branch into the feature branch and resolves the conflict textually. It does not capture resolutions that happen via rebase with manual conflict handling, which rewrite history and erase the merge commit. This is a lower-bound characterization.

**External validity â€” Public-GitHub scope.** Results should not be extrapolated to low-visibility or private repositories without care. Under `--pop-only`, the additional restriction to stars > 100 further biases toward popular, actively maintained projects.

**External validity â€” Agent mix.** Codex dominates AIDev by volume (roughly 87 % of all AIDev PRs). The cross-cutting stratification by agent is the primary means to detect and report this skew; per-agent results should be read with attention to sample sizes.

**Internal validity â€” Chronology completeness.** `extract_pr_chronology.py` relies on `refs/pull/*/head` being present on the remote at fetch time. GitHub garbage-collects these refs for PRs in very old/inactive repositories or when the PR branch has been force-deleted, in which case the chronology will carry no commits for that PR (recorded as `pr_ref_not_found` in `data/pr_chronology/errors.parquet`). These PRs are therefore excluded from Pass B. A stratified audit (by PR age and agent) is planned to quantify the fraction lost to this effect and to check that it does not correlate with conflict presence.

## 10. Implementation Decisions Left for Stage 0

These are intentionally deferred to be resolved empirically once the universe frame is built, rather than pre-committed:

- The exact disk budget and shard size will be tuned after measuring the actual average bare-clone size on a 100-repository sample.
- The agent-account signature map will be generated from `pull_request.parquet` grouped by `agent` and verified manually before Stage 6 runs. Entries with ambiguous human-vs-agent accounts (for example, a user account that is listed under multiple agents) will be flagged.
- If the replay volume in Stage 3 turns out to materially exceed the 5-day budget, a stratified sample (by `agent` Ă— top-N `language`) will be drawn at the PR level before Stage 3; the exact cutoff would be chosen to preserve statistical power for the per-stratum comparisons.
- The final merge audit (one-off check that PRâ†’base merges are almost always conflict-free) will be produced either as a separate Stage 3-audit pass or folded into Stage 3 with a flag, depending on incremental cost.
- **Scope-dependent sampling for Pass B.** Full AIDev has roughly 40x more PRs than AIDev-pop; the replay cost (Â§7) grows with the number of PRs that actually contain internal merges, which is unknown until after Pass A. A pragmatic plan is: (i) run Pass A end-to-end for full AIDev (network-bound, tractable over days); (ii) measure the distribution of "PRs with at least one 2-parent commit" per repository from the chronology output; (iii) if the total replay workload exceeds budget, draw a stratified sample per `agent Ă— top-N language` at the PR level before Pass B. The exact cutoff would be chosen to preserve statistical power for the per-stratum comparisons.
- **Note on earlier design.** An earlier revision of this plan pinned the scope to AIDev-pop because AIDev ships `pr_commits.parquet` only for the pop tier. A `--full-aidev` flag that tried to bridge the gap with a `git log --grep="pull request #N"` fallback was removed in the 2026-04-20 revision because the heuristic was circular (`Merge branch 'main' into <feature>` commits do not mention the PR number, while `Merge pull request #N` is the integration merge that must be excluded). The 2026-04-21 revision replaces that failed fallback with a proper enumeration via `refs/pull/*/head` (Â§5.1, Pass A).
