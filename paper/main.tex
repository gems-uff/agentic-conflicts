\begin{abstract}
AI-powered coding agents such as Claude Code, OpenAI Codex, GitHub Copilot, Cursor, and Devin are increasingly contributing to open-source software development through pull requests (PRs). When these AI-authored branches incorporate upstream changes via merge commits, textual merge conflicts can arise and must be resolved---either by the AI agent itself or by a human collaborator. Despite the growing adoption of agentic AI in software development, little is known about how merge conflicts are resolved in this context: which strategies are employed, who performs the resolution, and what factors influence the choice of strategy. In this paper, we present a large-scale empirical study of merge conflict resolution in AI-authored PRs, building upon the AgenticFlict dataset of 142,652 PRs opened by five AI agents across 59,412 repositories. By reproducing real merge commits found within PR branches using three-way \texttt{diff3} merging and comparing the conflicting regions against the actual resolved files, we classify each conflict chunk into one of eight resolution strategies (V1, V2, ConcatV1V2, ConcatV2V1, Combination, New~Code, None, Imprecise) and identify whether the resolver was an AI agent or a human developer. Our results show \TODO{summarize key findings}. We discuss the implications of these findings for the design of AI-assisted merge tools and for understanding the collaborative dynamics between AI agents and human developers.
\end{abstract}

The increasing adoption of AI-powered coding agents---such as Claude Code, OpenAI Codex, GitHub Copilot, Cursor, and Devin---represents a paradigm shift in software development workflows. These agents autonomously create branches, implement features, and open pull requests (PRs) in open-source repositories, contributing code alongside human developers.

When an AI-authored branch needs to incorporate upstream changes (e.g., via \texttt{git merge main}), textual merge conflicts may arise. These conflicts must be resolved before the merged code can be committed, and the resolution may be performed by the AI agent itself, by a human developer, or collaboratively.

Despite the growing scale of AI contributions---the AgenticFlict dataset~\cite{agenticflict2025} alone catalogues over 142,000 AI-authored PRs---the merge conflict resolution behavior of AI agents remains largely unexplored. Prior work on merge conflict resolution has focused exclusively on human developers~\cite{TODO-related-work}, leaving fundamental questions unanswered in the agentic AI context.

\subsection{PR Selection and Scope}
\label{sec:meth-selection}

Our study leverages the AgenticFlict dataset as its starting point. We consider all 142,652 PRs authored by AI agents for which both \texttt{base\_oid} and \texttt{head\_oid} are available. For each PR, we classify its outcome (\textit{merged}, \textit{closed\_unmerged}, or \textit{open}) based on the \texttt{merged\_at} and \texttt{gh\_state} fields.

\subsection{Identifying Merge Commits Within PR Branches}
\label{sec:meth-merges}

For each PR, we identify merge commits (commits with exactly two parents) that exist within the PR branch but not in the base branch:

\begin{verbatim}
git rev-list --merges base_oid..head_oid
\end{verbatim}

These are merges that the PR author (or collaborators) created to bring upstream changes into the feature branch---typically via \texttt{git merge main}. We exclude octopus merges (more than two parents) and do not consider the integration merge created by GitHub when the PR is accepted.

\TODO{Report the number of merge commits found and the fraction of PRs that contain at least one merge commit.}

\subsection{Reproducing Merges and Extracting Conflict Chunks}
\label{sec:meth-conflicts}

For each merge commit with parents $p_1$ and $p_2$, we compute the merge base ($b = \text{merge-base}(p_1, p_2)$) and identify files modified on both sides using \texttt{git merge-tree}. For each such file, we reproduce the three-way merge using:

\begin{verbatim}
git merge-file -p --diff3 p1_file base_file p2_file
\end{verbatim}

We parse the resulting diff3 output to extract individual conflict chunks, each delimited by \texttt{<<<<<<<}, \texttt{|||||||}, \texttt{=======}, and \texttt{>>>>>>>} markers. Each chunk produces three text regions: \textit{v1} (from $p_1$), \textit{base} (from $b$), and \textit{v2} (from $p_2$).

\subsection{Localizing the Resolution}
\label{sec:meth-resolution}

For each conflict chunk, we retrieve the resolved file from the merge commit itself via \texttt{git show \{merge\_sha\}:\{file\_path\}}. Since this file contains the entire contents (including non-conflicting regions and other resolutions), we isolate the specific resolution text for each chunk using contextual anchoring: we identify the unique prefix and suffix surrounding the conflict region in the merged output and use them to extract the corresponding slice from the resolved file.

\subsection{Classifying Resolution Strategies}
\label{sec:meth-strategies}

We classify each resolved chunk into one of eight resolution strategies using the \texttt{identify\_resolution} function, which compares the normalized resolution text against \textit{v1} and \textit{v2}:

\begin{itemize}
    \item \textbf{V1} / \textbf{V2}: The resolution is identical to one side of the conflict.
    \item \textbf{ConcatV1V2} / \textbf{ConcatV2V1}: The resolution is the concatenation of both sides, in the given order.
    \item \textbf{Combination}: The resolution contains elements from both sides but is not a simple concatenation.
    \item \textbf{New Code}: The resolution introduces content not present in either side.
    \item \textbf{None}: Both sides and the resolution are empty.
    \item \textbf{Imprecise}: The resolution could not be reliably classified.
\end{itemize}

For the statistical analysis in RQ4, we also define an aggregated four-level taxonomy: \textit{take-one-side} (V1, V2), \textit{combine} (ConcatV1V2, ConcatV2V1, Combination), \textit{introduce-new} (New Code), and \textit{trivial-or-other} (None, Imprecise).

\subsection{Identifying the Resolver}
\label{sec:meth-resolver}

We determine who resolved each conflict by analyzing the author and committer metadata of the merge commit. We classify the resolver as \textit{AI}, \textit{Human}, or \textit{Unknown} based on a curated list of known AI agent signatures (e.g., \texttt{claude[bot]}, \texttt{openai-codex[bot]}, \texttt{copilot[bot]}, \texttt{cursor-agent[bot]}, \texttt{devin-ai-integration[bot]}).

\subsection{Textual Similarity Metrics}
\label{sec:meth-similarity}

For each resolved chunk, we compute the Gestalt pattern matching (longest common subsequence) similarity between the resolution and each side of the conflict (\textit{sim\_res\_v1}, \textit{sim\_res\_v2}), as well as between \textit{v1} and \textit{v2} themselves (\textit{sim\_v1\_v2}).
